## Strategic Frame

The future organization is not a team of humans.
It is a system of humans plus many Synthetic Intelligence agents working across defined operating roles.

For IN$DEX, this is not a productivity improvement.
This is the architecture of a civilization.

SIINDEX should not help citizens one task at a time.
SIINDEX should orchestrate the right role, the right agent, the right workflow, the right memory, and the right approval path for every mission a citizen declares.

**The citizen brings the mission. SIINDEX assembles the sovereign team.**

## Core Doctrine

**IN$DEX cannot be run as a single chat, single app, single agent, or single workflow.**

**The civilization requires many operating roles.**

Some roles imagine. Some roles build. Some roles clean. Some roles grow. Some roles maintain. Some roles protect. Some roles review. Some roles remember.

**SIINDEX is the orchestration layer that routes work to the right role at the right time with the right context, tools, permissions, and safety gates.**

**No citizen should need to know which agent to call.**

## The Six-Archetype Operating Model

Every IN$DEX mission, product, campaign, citizen opportunity, automation, governance proposal, marketplace feature, or cultural asset workflow must be routed through one or more of six operating archetypes.

---

### 1. Prototype

**Used when IN$DEX needs new possibilities.**

The Prototyper generates. It does not decide. It opens space.

**Outputs:**
- Feature ideas and opportunity paths
- Campaign concepts and citizen business ideas
- App screen concepts and product experiments
- Trailer angles and marketplace offers
- Learning paths and cultural activation ideas

**Question:** *"What could this become?"*

**SIINDEX Prototype Agent rules:**
- Can: generate 5–20 possibilities in response to a mission
- Can: score possibilities by feasibility, cultural fit, and citizen impact
- Cannot: commit to building without citizen selection
- Cannot: override cultural review on cultural content concepts
- Requires: citizen selection before moving to Build
- Logs: options generated, option selected, reason for selection

---

### 2. Build

**Used when an idea needs to become usable.**

The Builder makes things real. It converts selection into output.

**Outputs:**
- MVP screen or page
- Marketplace listing
- Workflow or Mission Room
- Automation or prompt
- Demo, prototype, or build spec
- Income stream setup

**Question:** *"How do we make this real?"*

**SIINDEX Build Agent rules:**
- Can: create draft screens, listings, workflows, automations
- Can: generate build spec from citizen brief
- Cannot: publish without citizen approval
- Cannot: bypass cultural review on cultural assets
- Cannot: commit financial resources without explicit citizen authorisation
- Requires: citizen approval before any public-facing publish
- Logs: draft created, version history, citizen approval timestamp

---

### 3. Sweep

**Used when something exists but is messy, bloated, confusing, unsafe, or overbuilt.**

The Sweeper removes. It is the most underused role in every organization.

**Outputs:**
- Simplified UI and cleaner copy
- Reduced steps and removed features
- Fixed trust gaps and lower cognitive load
- Stronger security posture
- Fewer distractions and friction points
- Better cultural boundaries
- Cleaner data flows

**Question:** *"What should be removed, simplified, clarified, or protected?"*

**SIINDEX Sweep Agent rules:**
- Can: identify and flag complexity, friction, trust issues, and security gaps
- Can: propose removal or simplification with before/after comparison
- Cannot: delete or modify production assets without citizen approval
- Cannot: reduce cultural protection guardrails to simplify UX
- Requires: citizen review of all proposed removals before execution
- Logs: issues identified, proposed changes, citizen approval, final changes made

---

### 4. Grow

**Used when something works and needs stronger adoption, usefulness, trust, or economic impact.**

The Grower improves. It makes useful things reach more citizens.

**Outputs:**
- Onboarding improvements and retention loops
- Merchant growth paths and first-sale improvements
- Opportunity Feed placement and citizen education
- Useful return loops and trust signals
- Growth experiments and adoption analysis

**Question:** *"How does this become more useful to the right citizens?"*

**SIINDEX Grow Agent rules:**
- Can: analyse adoption data and identify friction points
- Can: generate improvement hypotheses and A/B test concepts
- Can: design first-sale paths and referral loops
- Cannot: promise income outcomes or guarantee results to citizens
- Cannot: use manipulative dark patterns (manufactured urgency without real scarcity, infinite scroll with no purpose, anxiety as retention)
- Requires: citizen consent for personalization signals
- Logs: growth actions, experiment results, adoption changes

---

### 5. Maintain

**Used when something must stay secure, reliable, compliant, fast, and trustworthy.**

The Maintainer protects. It is the role most organizations cut first and most regret later.

**Outputs:**
- Continuous monitoring and risk checks
- Security review and compliance audit
- Audit logs and uptime checks
- Data protection and recovery processes
- Governance guardrails
- Cultural protection checks
- Child safety enforcement

**Question:** *"How do we keep this safe, reliable, and sovereign?"*

**SIINDEX Maintain Agent rules:**
- Can: run automated checks and flag risks
- Can: pause high-risk actions for human review
- Can: generate security and compliance reports
- Cannot: override citizen decisions on risk acceptance without escalation
- Cannot: weaken cultural protection rules to improve performance
- Requires: human review for Black-level security events and cultural incidents
- Logs: all risk events, all checks, all escalations, all resolutions

---

### 6. Orchestrate

**Used always. By SIINDEX.**

The Orchestrator does not do the work. The Orchestrator decides who does the work, when, with what tools, and under what limits.

**Outputs:**
- Role selection and agent routing
- Approval gates and tool access decisions
- Escalation decisions and memory updates
- Decision Ledger entries
- Mission progress tracking
- Next best action for the citizen

**Question:** *"Which intelligence should act now, under what limits, with what review?"*

**SIINDEX Orchestrator rules:**
- Always active — every mission passes through Orchestrate first
- Routes to the minimum archetype needed (do not over-deploy)
- Maintains context across all roles in a mission
- Escalates to human review when: cultural, child safety, governance, financial risk, or security triggers are hit
- Updates memory after every mission milestone
- Creates a Decision Ledger entry for every orchestration decision
- Never acts outside declared permission scope
- Can pause, redirect, or close a mission at any time
- Citizen can override any orchestration decision (with log)

## Mission Rooms Upgrade — Agent Rooms

**This model upgrades Mission Rooms into Agent-Orchestrated Command Centres.**

A Mission Room should not only be a collaboration space. It should be the place where SIINDEX deploys the right agents, tracks their work, and surfaces decisions for citizen review.

**Example mission:**

A citizen says: *"I want to launch handmade Vanuatu print bundles for tourists."*

SIINDEX opens a Mission Room and deploys:

| Agent | Archetype | Task |
|---|---|---|
| Opportunity Agent | Prototype | Generates 10 possible offer structures |
| Cultural Steward Agent | Maintain | Checks cultural permission and attribution status |
| Sweep Agent | Sweep | Removes culturally risky or low-feasibility options |
| Builder Agent | Build | Creates the opportunity page and marketplace listing draft |
| Grow Agent | Grow | Designs first-sale path and hotel/tourism outreach strategy |
| Maintain Agent | Maintain | Sets up order tracking, customer issue flags, and trust review |
| Orchestrator (SIINDEX) | Orchestrate | Tracks all agents, requests citizen approval at each gate, logs to Decision Ledger |

**Citizen experience:** *"I bring the idea. SIINDEX brings the team."*

## Agent Declaration Standard

Every SIINDEX agent deployed in any Mission Room or automation must be declared with:

```
Agent Name: [name]
Archetype: [one or more of the six]
Can do: [specific permitted actions]
Cannot do: [explicit prohibitions]
Requires: [approval gates or human review triggers]
Logs: [what goes into the Decision Ledger]
```

**Example — Marketplace Listing Agent:**
```
Agent Name: Marketplace Listing Agent
Archetype: Builder / Sweeper
Can do: Draft listings, improve title, clarify buyer trust, suggest pricing range
Cannot do: Publish without approval, guarantee income, bypass cultural review
Requires: Citizen approval before publish
Logs: Listing changes, risk warnings, final approval timestamp
```

**Example — Cultural Steward Agent:**
```
Agent Name: Cultural Steward Agent
Archetype: Maintainer / Sweeper
Can do: Classify cultural asset, request permission, generate attribution
Cannot do: Approve sacred/restricted commercialisation alone
Requires: Elder/community review where indicated by Cultural Rights Graph
Logs: Cultural status classification, permission request, Decision Ledger entry
```

## The Internal Build Process — Six-Archetype Framework

For IN$DEX itself, every major feature should be routed through the six-archetype model before ship:

| Archetype | Question | For every feature |
|---|---|---|
| Prototype | What could this become? | Generate 5 versions or angles |
| Build | How do we make it real? | Build the MVP |
| Sweep | What should be removed or simplified? | Remove jargon, unnecessary steps, crypto complexity |
| Grow | How does this become more useful? | Improve completion rate and first useful action |
| Maintain | How do we keep it safe and sovereign? | Secure memory, consent, cultural boundaries, data privacy |
| Orchestrate | When is it ready, for whom, and what comes next? | SIINDEX decides: demo, user testing, or rebuild |
