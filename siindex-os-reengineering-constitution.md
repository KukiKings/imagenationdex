# SIINDEX Sovereign Operating-System Re-Engineering and Model Portfolio Constitution
> Status: CANONICAL. Adopted 18 Jul 2026 (Session 121 continued, Task #187).
> Source: full text supplied by AJ in chat, reproduced verbatim below.
> Governs: Prompt Intelligence, Mission Control, Mission Rooms, Background Work, App Factory, Build Integrity, Model Routing, Tool Authority, Work Receipts.

---

## Canonical purpose

SIINDEX should not automate broken habits one task at a time. SIINDEX should help citizens, businesses and institutions examine the full operating system behind their work. SIINDEX then proposes a better system. The citizen decides which changes to adopt.

This layer connects: Prompt Intelligence · Mission Control · Mission Rooms · Background Work · App Factory · Build Integrity · Model Routing · Tool Authority · Work Receipts.

---

## 1. Operating-System Re-Engineering

An operating system is the repeatable structure behind an objective. Examples: how Arthur manages the IN$DEX build; how a merchant handles orders; how a teacher plans lessons; how compliance reviews cases; how a citizen applies for work; how a corridor launches; how engineering releases software; how a family sends support.

SIINDEX should identify: inputs, decisions, tasks, people, tools, delays, approvals, costs, risks, evidence, failures, outputs, repeated work.

The goal is not more automation. The goal is a better outcome with less friction and stronger control.

---

## 2. Constitutional laws

**Law 53 — No System Redesign Without an Objective**
Open exploration is permitted inside a sandbox. Production redesign requires: defined objective, citizen, current process, known problems, constraints, authority, success measures, rollback.

**Law 54 — SIINDEX Proposes, the Citizen Governs**
SIINDEX may: challenge assumptions, identify waste, propose new workflows, compare alternatives, build prototypes, test ideas. SIINDEX does not gain the right to choose every business, financial, legal or personal direction.

**Law 55 — No "Do Everything" Authority**
A phrase such as "SIINDEX, do it all" does not grant unrestricted access. SIINDEX must resolve: what work? which data? which tools? which accounts? which budget? which approvals? which deadline? which prohibited actions?

**Law 56 — Models Do Not Handle Permanent Secrets**
Models must not receive: permanent API keys, seed phrases, passwords, signing keys, database service-role keys, treasury credentials, identity-recovery secrets. A secrets broker provides narrow, short-lived capabilities to approved tools.

**Law 57 — Model Intelligence and Authority Are Separate**
A more capable model does not receive: more data, more tools, more spending authority, more execution rights, more memory. Authority comes from the runtime and Work Mandate.

**Law 58 — Exploration and Production Are Separate**
Creative exploration may be open-ended. Production implementation requires: specification, tests, security, evidence, review, deployment gate, rollback.

**Law 59 — Model Substitution Must Be Visible**
If a task is routed to another model, record: requested model, actual model, reason, pricing difference, data-retention difference, capability difference, evaluation result.

**Law 60 — Every Redesign Must Be Measured**
A proposed operating system is not better because it sounds sophisticated. Measure: time, cost, error rate, completion, citizen comprehension, human workload, risk, recovery, repeat value, provider dependence.

## 3. Operating-System Audit

The citizen says: "SIINDEX, review how I run this." SIINDEX maps:

- **Objective** — what result is expected?
- **Current workflow** — what happens now?
- **Actors** — who participates?
- **Systems** — which applications, providers and records are used?
- **Friction** — where does work stall?
- **Duplication** — which information is entered more than once?
- **Failure** — what breaks?
- **Risk** — where can harm, fraud, data loss or unauthorised action occur?
- **Authority** — which decisions belong to the citizen or qualified human?
- **Evidence** — how is completion proved?
- **Exit** — how can the citizen leave or replace a provider?

---

## 4. Challenge Council

Before redesigning a workflow, run six reviewers:

1. **Citizen Advocate** — does the new system improve the citizen's result?
2. **Operations Reviewer** — does it remove delay and duplication?
3. **Security Reviewer** — does it increase attack surface?
4. **Sovereignty Reviewer** — does it increase provider or model dependence?
5. **Human Responsibility Reviewer** — which decisions must remain human?
6. **Commercial Reviewer** — does the system have a sustainable delivery model?

The council should produce disagreement, not forced consensus.

---

## 5. Four-stage engineering loop

Replace uncontrolled prompt-build cycles with:

**SPECIFY** — Objective Contract, system map, data contract, authority map, threat model, acceptance criteria, rollback.

**BUILD** — Implement inside branch, sandbox, preview environment, isolated data, bounded tool permissions.

**VERIFY** — Functional tests, security tests, authorisation tests, accessibility tests, data-integrity tests, failure tests, visual checks, cost checks.

**RELEASE** — Review, signed release, staged deployment, monitoring, rollback, Release Integrity Receipt.

The loop repeats only when tests provide a reason.

---

## 6. Model Portfolio Router

IN$DEX should not depend on one model. Maintain approved model classes:

- **Frontier strategy model** — system redesign, architecture, complex research, difficult diagnosis, adversarial review, high-context synthesis.
- **Balanced production model** — implementation, ordinary analysis, document production, workflow execution, coding with clear specifications.
- **Fast operational model** — classification, extraction, simple transformation, routing, formatting, repetitive low-risk work.
- **Local or private model** — sensitive processing, offline work, private classification, low-connectivity environments, restricted data.

Model selection considers: capability, evaluation history, cost, latency, context, privacy, data retention, region, availability, safety behaviour, tool performance.

---

## 7. Model Passport

Every approved model records: provider, model, version, release date, retirement date, context, output limit, price, latency, supported tools, supported modalities, retention, training policy, region, safety routing, known failures, benchmark results, approved tasks, prohibited tasks, fallback, last review.

Promotional access does not determine system architecture.

---

## 8. Strategic and execution separation

Use frontier models where their added capability is proven. Do not use them for every repetitive operation. Example:

1. Frontier model analyses an IN$DEX corridor.
2. Balanced model creates implementation specifications.
3. Coding model implements approved changes.
4. Fast model runs routine classification.
5. Independent model reviews the result.
6. Deterministic tests make the release decision.

The frontier model does not approve its own work.

---

## 9. Creativity Sandbox

The citizen may say: "SIINDEX, challenge how we designed this." Inside the sandbox, SIINDEX may: propose unconventional ideas, combine systems, identify hidden assumptions, generate alternatives, build disposable prototypes, compare future scenarios.

Sandbox restrictions: no production writes, no real payments, no real citizen data, no permanent secrets, no public publishing, no governance execution, no provider commitments.

Promising ideas move into the formal specification process.

## 10. Mission Control

Mission Control should not be a static grid of hundreds of screens. It should be a live interface showing: what changed, what needs attention, what failed, what is waiting for approval, what is stale, what threatens the objective, what SIINDEX recommends, what evidence supports each claim.

The interface should be generated from active Mission Workspaces.

## 11. Personal and citizen operating systems

- **Founder OS** — build briefing, decisions, partner pipeline, compliance, security, finance, launch readiness.
- **Merchant OS** — orders, messages, stock, invoices, payouts, customer follow-up, disputes.
- **Citizen Opportunity OS** — skills, learning, work, applications, income paths, evidence, next action.
- **Teacher OS** — curriculum, lesson plans, assessments, parent updates, workload, student safeguards.
- **Family Support OS** — remittance, bills, sponsored services, receipts, emergency support.

Each operating system remains citizen-controlled and portable.

---

## 12. Browser and computer-use boundary

**Allowed low-risk examples** — gather public information, compare published prices, fill a draft form, prepare a checkout, download an authorised document, test a preview environment.

**Approval required before** — account creation, accepting terms, submitting personal data, purchasing, sending, publishing, changing settings, granting permissions, deploying to production.

**Blocked without exceptional authority** — retrieving permanent secrets, changing treasury credentials, disabling security, force-pushing protected branches, approving its own access, modifying identity recovery.

---

## 13. Secrets Broker

Tools request capabilities from the broker. A capability contains: tool, account, purpose, action, resource, environment, expiry, maximum uses, maximum value, approval, correlation ID.

The model sees: "Deployment permission granted for preview project until 3:30 pm." The model does not see the permanent platform credential.

---

## 14. Reverse Prompting, corrected

The safe form: "Review this operating system. Identify the strongest opportunities, hidden assumptions, risks and alternatives. Do not implement changes. Rank your recommendations by citizen value, evidence, cost, risk and reversibility."

Do not use: "Choose everything for me and do whatever you think is best."

---

## 15. System-redesign deliverable

Every redesign should produce: current-state map, problem statement, citizen objective, proposed system, alternatives, retained human responsibilities, data flow, authority map, risk analysis, provider dependencies, cost estimate, implementation sequence, tests, rollback, success measures.

---

## 16. Model cost governor

Every workflow records: model, input tokens, output tokens, cache usage, tool cost, total cost, value created, retries, failures, cheaper eligible model.

Set: per-run limits, daily limits, workspace limits, citizen limits, warning thresholds, emergency stop.

Expensive models should be used where evaluations show measurable added value.

---

## 17. Model continuity

When a model becomes unavailable, becomes too expensive, changes behaviour, changes retention, fails evaluation, is restricted by region, or is retired — the workflow should move to an approved alternative. The Objective Contract, Work Mandate, receipts and data remain portable.

## 20. Acceptance tests

- **Objective test** — an open-ended exploration cannot enter production without an Objective Contract.
- **Authority test** — a stronger model receives no additional tool authority.
- **Secret test** — the model cannot retrieve a permanent API key.
- **Browser test** — the browser prepares an account application but cannot submit it without approval.
- **Deployment test** — the model can deploy to preview but not production without the Release Gate.
- **Model substitution test** — the actual model is recorded when the requested model is replaced.
- **Cost test** — the workflow routes routine work to the lower-cost approved model.
- **Failure test** — a model outage triggers an approved fallback without losing task state.
- **Review test** — the model that builds a change cannot be the sole evaluator.
- **Sovereignty test** — the workflow continues after replacing the model provider.

---

## Final doctrine

A frontier model is not authority. Original thinking is not evidence. Open exploration is not production. A browser agent is not an administrator. A model must not hold permanent secrets. The most expensive model is not always the right model.

The model proposes. The citizen chooses. The runtime grants bounded authority. Specialist models perform the work. Independent tests verify the result. IN$DEX records the model, cost, evidence and outcome. The objective survives the model. The citizen remains sovereign.
