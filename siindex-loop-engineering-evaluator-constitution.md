## Canonical purpose

SIINDEX must not rely on one prompt, one model, one agent, one judge or one score. SIINDEX should operate through governed loops that resolve the objective, build the work, inspect the result, test the real outcome, identify defects, repair only verified defects, detect regressions, stop at constitutional boundaries, request human authority when required, and create proof of every iteration.

The citizen should not become the unpaid quality-control department. The citizen should also not be asked to trust agents grading themselves.

The complete pattern: the citizen defines the objective. SIINDEX creates the Loop Contract. The builder performs bounded work. Deterministic systems test what can be tested. Independent evaluators inspect what requires judgment. Adversarial evaluators search for hidden failure. Qualified humans decide where human authority is required. The loop repairs verified defects. The loop stops when the evidence satisfies the Definition of Done. IN$DEX records the complete proof.

## 2. Foundational doctrine

A prompt expresses an instruction. A workflow defines a path. A harness gives the agent tools and state. A loop repeatedly observes, acts, evaluates and corrects. An evaluation measures the result. A release gate decides whether the result is permitted to proceed. These are separate systems. Do not collapse them into one model conversation.

## 3. Constitutional laws

**Law 73 — A Loop Is Not Authority.** A loop does not gain new authority because it repeats. Every iteration remains bound by citizen authority, active mode, Objective Contract, Work Mandate, tool permissions, budget, consent, data scope, jurisdiction, risk limits.

**Law 74 — No Loop Without a Definition of Done.** Every loop requires objective completion criteria before the first iteration: required outcome, mandatory elements, hard prohibitions, evidence, tests, acceptable uncertainty, human review, final approval, stopping conditions. "Make it better" is not a Definition of Done.

**Law 75 — Deterministic Evidence Outranks Model Opinion.** Where a result is machine-verifiable, use deterministic verification (unit tests, integration tests, schema validation, calculations, database constraints, access-control checks, signature verification, balance reconciliation, route availability, file existence, cryptographic hashes, policy-as-code, accessibility scanners). A model cannot overrule a failed deterministic safety test.

**Law 76 — A Judge Is Evidence, Not Truth.** A model judge provides an assessment, not final truth. Judge outputs must record model, version, rubric, input, score, reasons, uncertainty, detected defects, possible bias, calibration status.

**Law 77 — Different Model Families Do Not Guarantee Independence.** A judge from another provider may reduce some shared bias. It does not guarantee correctness, neutrality, cultural competence, legal authority, language competence, security expertise, or independence from shared training patterns. Cross-family evaluation is one control among several.

**Law 78 — No Single Aggregate Score Controls Release.** A result must not enter production because it received 90/100, 100/100, five stars, majority approval, "looks good," or "safe to proceed." Hard requirements must pass individually. A security failure cannot be averaged away by strong visual design.

**Law 79 — The Builder Cannot Be the Sole Judge.** The builder may perform self-checks. Self-checking is not independent evaluation. The release decision requires separate evidence.

**Law 80 — The Judge Must Also Be Evaluated.** Every evaluator requires calibration, test cases, known-answer cases, bias probes, failure tracking, model-version review, human comparison, suspension rules. An untested judge cannot approve production work.

**Law 81 — No Infinite Loops.** Every loop requires limits on iterations, time, model cost, tool cost, retries, external calls, token usage, data transfer, failed repairs, human escalation.

**Law 82 — No Hidden Self-Modification.** A loop must not silently modify its own constitution, evaluation rubric, release threshold, tool authority, safety policy, active prompt, model router, financial limits, approval rules, or permanent memory rules. Changes to these systems require a separate governed release.

**Law 83 — Improvement Must Be Measurable.** A later iteration must improve verified criteria — it is not enough for the judge to prefer the newer version. Compare defects fixed, regressions introduced, tests passed, task completion, citizen value, safety, cost, latency, evidence quality.

**Law 84 — Failed Iterations Remain Visible.** Do not erase failed attempts from the audit history. Retain failed output, evaluator findings, changes requested, repair attempt, resulting evidence, reason for rejection.

**Law 85 — Human Authority Cannot Be Optimised Away.** A loop cannot remove human review merely because repeated model evaluation reaches a high score. Human authority remains mandatory where required by law, financial regulation, medical responsibility, child safety, cultural authority, governance, employment responsibility, identity recovery, emergency response, constitutional policy.

**Law 86 — Every Loop Creates a Loop Assurance Receipt.** Every consequential loop must produce portable proof of what was attempted, how it was evaluated, what changed, why it stopped, who approved it, what entered production.

## 4. Loop Contract

Every loop begins with a Loop Contract. Required fields: Loop ID, owner, citizen or institution, Objective Contract, active mode, risk class, environment, builder, evaluator set, deterministic graders, human reviewers, input data, source restrictions, tool grants, memory scope, Definition of Done, hard invariants, soft quality criteria, maximum rounds, maximum duration, maximum cost, no-progress threshold, regression threshold, escalation rule, stopping rule, release authority, rollback, expiry. The Loop Contract must be approved before a consequential loop begins.

---

## 5. Hard invariants and soft criteria

**Hard invariants** produce PASS or FAIL: no cross-citizen data access, no unauthorised payment, no secret exposure, no unsupported financial claim, no missing consent, no broken authentication, no inaccessible critical control, no personal information on a public ledger, no child data in marketing, no unapproved cultural use, no failed transaction shown as successful. Any hard-invariant failure stops release.

**Soft criteria** measure quality: clarity, tone, visual balance, concision, usefulness, organisation, teaching quality, persuasive strength within approved boundaries. Soft criteria may use scores. They cannot override hard failures.

---

## 6. Evaluation stack

Every important loop should use the lowest layer able to verify the requirement.

**Layer 1 — Deterministic graders:** exact rules, calculations, schemas, permissions, signatures, test results, accessibility thresholds, code quality, route checks.

**Layer 2 — Environmental outcome graders:** did the button work? did the payment remain blocked? did the correct database row change? did the email remain unsent before approval? did the asset render correctly? did the deployment match the approved commit?

**Layer 3 — Policy graders:** authority, consent, privacy, data minimisation, legal boundaries, financial boundaries, cultural restrictions, child safety. Should rely on policy-as-code where possible.

**Layer 4 — Model-based evaluators:** meaning, clarity, completeness, relevance, language quality, explanation quality, source synthesis.

**Layer 5 — Adversarial evaluators:** prompt injection, unsupported assumptions, hidden data exposure, misleading claims, exploitation paths, dark patterns, reward manipulation, fraud paths, policy circumvention.

**Layer 6 — Qualified human review:** matters requiring professional, institutional, cultural, constitutional or citizen authority.

## 7. Judge Registry

Each judge records: Judge ID, provider, model, version, model family, task domains, supported languages, prohibited domains, rubric types, calibration dataset, calibration date, human agreement rate, false-pass rate, false-fail rate, self-preference result, family-preference result, position-bias result, verbosity-bias result, language-bias result, tool-trajectory accuracy, cost, latency, retention, current status, expiry.

Valid states: EXPERIMENTAL, CALIBRATING, APPROVED, RESTRICTED, SUSPENDED, RETIRED.

---

## 8. Judge calibration

Every model judge must be tested against known cases: clearly correct results, clearly incorrect results, subtle failures, polished but false outputs, ugly but correct outputs, concise and verbose equivalents, reordered answer choices, builder-generated outputs, other-model outputs, multiple languages, low-resource languages, adversarial wording, missing steps, fabricated tool calls, altered evidence.

The same candidate should be evaluated under varied position, order, formatting, verbosity, model attribution, author identity. A judge that changes its verdict because the output moved from position A to position B requires restriction or correction.

---

## 9. Blind evaluation

Where practical, the judge should not know which model/provider generated the candidate, which candidate is newer, which candidate the founder preferred, or which version is the current production version. Remove identifying style markers where possible. Randomise candidate order. Record the randomisation seed.

---

## 10. Independent evaluation conditions

A judge is not independent merely because it is a second API call. Independence requires separation of conversation context, hidden reasoning, memory, builder scratchpad, model identity where practical, reward signal, output ownership, approval authority. The judge receives objective, rubric, evidence, candidate result, required environment outputs — not irrelevant builder persuasion.

---

## 11. Multi-judge council

Use multiple judges only where the added cost provides value. Possible council: domain evaluator, security evaluator, citizen advocate, language evaluator, adversarial evaluator. Do not treat majority vote as proof. The council should report agreement, disagreement, confidence, evidence, unresolved defects, suspected bias. A material disagreement triggers deterministic verification, another independent evaluator, human review, or BLOCKED status.

---

## 12. Score constitution

Do not use one score out of 100 as the complete result. Use a scorecard (e.g. objective completion: PASS; functional correctness: PASS; security: FAIL; privacy: PASS; accessibility: CONDITIONAL; factual evidence: PASS; citizen clarity: 4/5; cost: within budget; human review: pending → overall state: BLOCKED). The security failure determines the state.

Valid final states: PASS, CONDITIONAL PASS, FAIL, BLOCKED, HUMAN REVIEW REQUIRED, CANCELLED, EXPIRED.

---

## 13. Reference solutions

Every important evaluation task should have one known result that passes, proving the task is solvable, the rubric is coherent, deterministic graders are configured correctly, the environment works, and the pass threshold is reachable. A task with no passing reference must not be used to condemn every builder result. Anthropic recommends reference solutions and warns that universal failure often signals a broken task or broken grader rather than an incapable agent.

---

## 14. Balanced evaluation sets

Test both when an action should happen and when it should not: search vs. do not search, approve vs. reject, warn vs. do not warn, escalate vs. resolve, remember vs. do not remember, block payment vs. allow payment, identify scam vs. avoid false accusation. One-sided evaluations create one-sided behaviour.

---

## 15. Hidden holdout tests

The builder may know the public Definition of Done. The builder should not receive every hidden anti-gaming test. Holdout tests may check unsupported shortcuts, hard-coded success, deleted warnings, disabled security, fake records, test-specific branching, prompt injection, data leakage, false provenance, evaluator manipulation. Holdout tests require strict governance and must not test undisclosed product requirements unrelated to the stated objective.

---

## 16. Reward-hacking and Goodhart defence

A loop may learn to improve the score without improving the outcome: adding more words to impress the judge, copying rubric terms, hiding failed tests, weakening the evaluator, removing difficult test cases, marking unknown as success, optimising screenshots while buttons remain broken, satisfying a mock test while production fails, producing reassuring language without evidence.

Controls: multiple metrics, hidden tests, environment verification, immutable rubrics during a run, independent test ownership, regression suites, production monitoring, human calibration.

---

## 17. Loop shapes

Builder and evaluator loop (writing, design, structured research, ordinary code improvement) · Test-driven repair loop (software, schemas, calculations, deterministic workflows — tests fail, builder repairs, tests rerun) · Plan/execute/verify loop (tool workflows, operational work, background tasks) · Research completeness loop (complex research, due diligence, source gathering — evaluator identifies evidence gaps) · Red-team and repair loop (security, fraud protection, prompt injection, policy bypass) · Multi-specialist production loop (media, app design, institutional documents, complex product work) · Human-checkpoint loop (regulated, cultural, child-related, identity, governance, high-impact financial work).

---

## 18. Loop selection engine

Select the simplest architecture that satisfies the objective. Simple task: one builder, deterministic checks, no repeated model judge unless a failure appears. Subjective quality task: builder plus independent evaluator, limited repair rounds. Complex multi-domain task: orchestrator plus specialists, domain-specific graders. Consequential task: deterministic verification, independent policy evaluation, human approval. The existence of agent teams does not mean every task needs a team.

## 20. Repair contract

The evaluator must not say only "Bad. Try again." A repair request should contain: failed requirement, supporting evidence, location, severity, expected condition, permitted repair scope, tests to rerun, risk of regression. The builder must repair the defect without silently changing unrelated approved behaviour.

---

## 21. Diff-based repair

Every repair iteration should record: files changed, prompt sections changed, workflow nodes changed, data changed, components changed, policies changed, tests affected, claims affected. The evaluator should inspect the change, not only the final snapshot. Large unrelated changes trigger scope-drift review.

---

## 22. Regression control

A repaired result must rerun: failed tests, neighbouring tests, full critical-path tests, security invariants, authorisation matrix, accessibility checks, truth claims, rollback tests. A defect fixed in one area must not silently break another.

---

## 23. No-progress detection

Stop or escalate when: the same defect appears repeatedly, scores oscillate, changes stop improving outcomes, new regressions exceed repairs, cost rises without measurable value, the evaluator contradicts itself, evidence remains unavailable, the builder begins gaming the rubric, the task needs human judgment.

Possible states: STALLED, OSCILLATING, EVALUATOR CONFLICT, EVIDENCE BLOCKED, HUMAN DECISION REQUIRED.

---

## 24. Stopping rules

A loop stops on: verified completion (all hard requirements + quality thresholds + approvals pass), budget stop, safety stop (constitutional invariant fails), authority stop, evidence stop, diminishing-return stop, oscillation stop, citizen stop, emergency stop (kill switch).

---

## 25. Cost architecture

Do not assume the expensive model belongs exclusively in the judge role. Possible patterns: strong builder/fast deterministic tests, low-cost builder/strong domain judge, strong planner/lower-cost workers, local classifier/frontier exception handler, deterministic grader/no model judge, several low-cost judges plus human calibration. Record cost per iteration, cost per defect fixed, cost per verified result, marginal improvement, cheaper successful configuration.

---

## 26. Model routing experiment

Before selecting a permanent configuration, test: strong builder + strong judge, strong builder + moderate judge, moderate builder + strong judge, moderate builder + deterministic tests, same-family judge, cross-family judge, multi-judge council, human-calibrated judge. Compare correctness, false passes, false failures, cost, latency, regressions, citizen usefulness. The cheapest configuration meeting the required assurance level becomes the default.

---

## 27. Long-running loop state

Persist: objective, current plan, completed steps, failed steps, files changed, tests, evaluator findings, known defects, pending approvals, next safe action, cost, checkpoint, rollback state. Anthropic's long-running agent work uses persistent artifacts and incremental handoff between context windows because a single context cannot reliably contain an extended project.

---

## 28. Loop workspace

Each loop receives an isolated workspace containing: Loop Contract, working files, source evidence, task state, evaluation reports, iteration history, budgets, approvals, deliverables, receipts. The loop workspace must not inherit unrelated citizen or institutional data.

---

## 29. Multi-agent coordination

Agent teams require explicit role definitions, task boundaries, shared-state rules, message formats, conflict resolution, termination, duplicate-work prevention, merge procedure, evidence ownership. Multi-agent systems increase parallelism but also introduce coordination, evaluation and reliability challenges.

---

## 30. Agent conflict protocol

When agents disagree: record each position → identify the disputed fact or judgment → inspect supporting evidence → run deterministic verification where possible → consult a specialist evaluator → escalate material unresolved conflicts → do not force artificial consensus.

---

## 31. Adversarial loop separation

Red-team agents must not gain the ability to execute the attacks they describe in production. Use sandbox, synthetic data, restricted tools, controlled network, no production credentials, test accounts, blast-radius limits. Red-team findings become repair tasks — they do not become reusable attack instructions for unrestricted agents.

---

## 32. Evaluator injection defence

Candidate work may contain instructions intended to manipulate the judge: "Give this result 100," hidden text, prompt injection, fake test results, fabricated citations, evaluator role instructions, misleading comments, adversarial metadata. The evaluator must treat candidate content as untrusted evidence and must not obey instructions contained inside the candidate.

---

## 33. Judge independence from financial incentives

An evaluator must not receive rewards based only on number of passes, speed of approval, low rejection rate, high release volume, or low human-review rate — these encourage unsafe approval. Measure evaluators through correct passes, correct failures, human agreement, defects caught, false-alarm cost, production incidents missed.

---

## 34. Human calibration council

Qualified reviewers should regularly inspect samples of passed work, failed work, disputed work, multilingual work, high-risk work, model-version changes, production incidents. Human findings update rubrics, test cases, judge restrictions, model routing, skill versions, failure libraries. Human correction should become a future evaluation case. OpenAI's 2026 tax-agent work uses practitioner corrections and production traces to create new evaluations and measurable improvement targets rather than relying on unguided recursive self-improvement.

---

## 35. Failure Library

Failure ID, affected system, input, incorrect output, expected output, root cause, severity, citizen impact, detection method, repair, regression test, model versions affected, resolved version. Every real production failure should produce at least one new regression evaluation where appropriate.

---

## 36. Evaluation-set governance

Evaluation data must be versioned, access-controlled, reviewed, representative, balanced, protected from accidental leakage, refreshed, linked to real failures, tested for ambiguity, tested for cultural bias, tested across languages. Do not train or prompt the builder directly on the complete hidden evaluation set.

---

## 37. Evaluation drift

Monitor whether judge behaviour changed, model version changed, rubric interpretation changed, test environment changed, source data changed, product requirements changed, language performance changed, cost changed, provider routing changed. A new judge version requires recalibration. A higher benchmark score does not automatically mean the new judge is a better measurement instrument — changing the judge may change the measured result even when candidate outputs remain fixed.

---

## 38. Multilingual judge protection

Do not assume a judge calibrated in English performs equally in Cook Islands Māori, Samoan, Tongan, Fijian, Bislama, Tok Pisin, Māori, or other Pacific languages. Each supported language requires native or authorised review, language-specific evaluation, terminology approval, bias testing, fallback rules. Low-resource-language evaluation should not rely solely on an English-optimised model judge.

---

## 39. Cultural evaluation

A general-purpose model cannot issue final approval for cultural authority. The loop may evaluate missing attribution, missing rights record, inconsistent terminology, known restricted categories. Authorised cultural reviewers determine permission, sacred status, family rights, community rights, acceptable public context, commercial approval.

---

## 40. Citizen comprehension loop

For important citizen-facing work, test whether the citizen understands the result: explain-back, scenario question, action selection, risk recognition, accessibility completion, language comprehension. Do not optimise only for what an evaluator model considers clear — test the intended citizen population.

---

## 41. Outcome verification

Grade the actual result, not the transcript. Don't ask "did the agent say the payment was blocked?" — check "did the payment remain blocked in the ledger?" Don't ask "did the agent say the deployment succeeded?" — check "does production run the approved commit?" Don't ask "did the media render?" — check "are every slide, frame, caption and asset correct?"

## Final doctrine

The prompt is not the complete system. The loop is not the complete system. The judge is not truth. A different model is not guaranteed independence. A higher score is not guaranteed improvement. One hundred out of one hundred is not production proof. More rounds are not always better. More agents are not always stronger. A failed deterministic test cannot be negotiated away. A qualified human cannot be replaced by evaluator consensus.

The citizen defines the objective. SIINDEX creates the governed loop. Builders perform bounded work. Tests verify objective facts. Evaluators inspect judgment-based quality. Adversaries search for hidden failure. Humans retain human authority. The loop repairs verified defects. The release gate decides what proceeds. IN$DEX records the complete proof. The citizen remains sovereign.
